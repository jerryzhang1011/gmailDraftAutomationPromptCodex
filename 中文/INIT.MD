# Gmail 回复智能系统一次性 Bootstrap 规范

## 状态
- 本文件**仅用于 bootstrap**。
- 使用此文件来初始化项目；仅当用户明确要求重新按照 `INIT.MD` 执行时，才能再次运行初始化或补充流程。
- 请**不要**将此文件纳入日常的起草工作流。
- 两份目标文件仅仅在磁盘上存在，**不**构成完成信号。完成的定义基于文件内容（参见 `文件状态判定`），而不是文件是否存在。

## 目标
你正在为一个业务邮箱初始化一个 Gmail 起草项目。你的任务是通过 [@gmail](plugin://gmail@openai-curated) 连接器研究历史 Gmail 对话，并准确生成以下**两份**长期知识文件：

1. `COMPANY_BUSINESS_KNOWLEDGE.MD`
2. `EMAIL_REPLY_STYLE_GUIDE.MD`

这两份文件将成为未来草稿生成过程中**唯一**的持久参考文档。它们必须详尽、一致、基于证据，并且便于人工后续手动扩展。

## 不可妥协的规则
- 仅起草，不发送邮件。
- 不得归档、删除、重新打标签，或以任何方式改变邮箱状态。
- 不得编造价格、条款、政策、周转时间、业务范围或承诺。
- 不得将邮件正文原文复制进知识文件。
- 在能够提取出持久规则或模式时，不得仅写模糊的总结。
- 不得收录一次性的琐碎信息，除非其很可能影响未来的回复。
- 不得覆盖目标文件中由人工维护的章节。
- 除上述两份文件外，不得创建任何其他输出文件。
- 如果证据不足、互相矛盾或不完整，应记录在未决或低置信度的章节中，而不是将其规范化为确定规则。

## 必需输入
- 可通过 Gmail 连接器访问的历史 Gmail 对话。
- 两份目标文件（如已存在）。它们可能是空模板、部分填充，或已完全填充。

## 文件状态判定
在开始任何其他工作之前，请打开两份目标文件，并将**每个文件**归入以下三种状态之一。该判定应**逐个文件**进行，而不是把两份文件作为一个整体判定。

**状态 A：缺失（Missing）**。
- 文件在磁盘上不存在。
- 操作：按所需的章节顺序与 schema 创建该文件，然后执行完整的 bootstrap。

**状态 B：模板 / 未初始化（Template / Uninitialized）**。
- 文件已存在，但 agent 生成的部分尚未被填充。
- 只要满足以下任一条件，该文件就**必须**被视为 Template：
  - 多条 agent 生成条目仍然含有占位措辞，例如 `Not initialized yet`、`None yet`、`No known unknowns recorded yet`、`No evidence index entries yet`。
  - 所有 agent 生成条目的 `Confidence` 都是 `Low`，同时 source evidence 为空或为 `None yet`。
  - `<!-- AGENT_GENERATED_START -->` 与 `<!-- AGENT_GENERATED_END -->` 之间的内容仅包含样板说明，没有任何被提取出的规则。
- 如果 Human Overrides 章节仅含占位行，例如 `No human overrides yet`、`rule 1`、`rule 2`、`...`，则**不算**已被填充，该文件仍属于模板。
- 操作：按**全新 bootstrap** 对待。你**必须**继续基于 Gmail 证据将 agent 生成章节完整填充。不要停下，不要请示许可，也不要把该文件汇报为已初始化。

**状态 C：已填充（Populated）**。
- 文件已存在，agent 生成部分已经包含真实的、有证据支撑的规则，带有具体细节、在合理处非 `Low` 的置信度，以及具体的 `Source Evidence` 引用。
- 操作：**不得**跳过工作。仍需按照 `Bootstrap 范围` 与 `最低审阅标准` 进行完整的邮箱扫描，然后执行补充（supplementation）流程：
  - 补入尚未被记录的、新发现的持久规则。
  - 当新会话支持已有规则时，提升其置信度或补强其证据。
  - 将真正新出现的冲突记入 `Known Unknowns`。
  - 除非新证据明确与现有 agent 生成规则矛盾，否则不得重写、重排序或删除它们；在矛盾的情况下，应将有争议的规则移入 `Known Unknowns`，并同时记录双方证据。
  - 无论处于何种状态，都**绝不**改动 Human Overrides 的内容。

如果两份文件处于不同状态，请**分别**按各自的状态独立处理。

## 必需输出
准确地生成或刷新以下**两份**文件：

1. `COMPANY_BUSINESS_KNOWLEDGE.MD`
2. `EMAIL_REPLY_STYLE_GUIDE.MD`

本次 bootstrap 运行不得输出其他任何文件。

## Bootstrap 范围
研究来自 Primary 邮箱分类中的、信号丰富的客户、潜在客户、潜在项目、合作伙伴、供应商或业务运营对话。优先选取那些包含以下具体信息的会话：

- 公司销售什么
- 服务或产品如何打包
- 如何解释定价
- 哪些内容是包含的、哪些是不包含的
- 时间节点、交付物或流程步骤如何被描述
- 公司会拒绝什么、升级什么，或视作例外处理
- 回复是如何遣词、组织结构、软化、澄清和结尾的

请排除或大幅降低优先级的内容包括：

- 订阅通讯（Newsletter）
- 收据
- 密码重置邮件
- 营销群发
- 日历通知干扰
- 自动生成的警报
- 发给自己的个人备注
- 没有明显业务或风格信号的会话

## Gmail 工作方法
在使用 Gmail 连接器时应保持克制且系统化。

1. 构建一个可能高信号的 Primary 邮件短名单。
2. 在提取规则之前，将有价值的消息展开为完整会话。
3. 阅读足够多的会话，以捕捉稳定且反复出现的模式，而不仅仅是零散示例。
4. 仅当后续再阅读的会话多为重复、不再贡献新的持久业务规则或风格模式时，方可停止。

## 会话选择启发式
优先选择满足以下一项或多项条件的会话：

- 客户明确询问价格、范围、时间节点、交付物或流程的问题
- 涉及议价、报价、对接、跟进或排期的内容
- 同一种公司解释在多个会话中反复出现
- 看起来打磨良好、具有代表性、可复用的高质量公司回复
- 能体现语言切换处理方式的多语对话
- 能揭示政策边界的升级或例外处理情形

在以下情况下，**不要**将会话视为权威证据：

- 相关答复从未实际发出
- 会话过于碎片化，不足以支撑一条持久规则
- 语言异常情绪化、临时凑合，或与整体语料明显不一致
- 该消息看起来像是个人备注，而非业务沟通

## 最低审阅标准
- 审阅足够多的合格会话，以捕捉反复出现的业务知识与写作习惯。
- 覆盖面应横跨常见的客户意图，而不只是某一狭窄话题。
- 如果合格会话少于 30 个，则应尽可能审阅你能找到的所有合格会话。
- 如果合格会话超过 30 个，则应持续审阅，直到具备良好的覆盖面，且连续 10 个新增的合格会话都不再产生实质性新规则为止。

## 提取标准
每一条提取出的条目都必须是**持久的、可复用的、具体的**。

良好的提取目标示例：
- "服务 X 通常在一次简短的发现/需求沟通步骤之后报价。"
- "价格以定制报价、区间报价或套餐报价的方式呈现，而不是固定价格。"
- "公司通常在回复的第一句话确认客户的问题。"
- "回复通常以一句简短的下一步邀请加上具名落款结尾。"

不良的提取目标示例：
- "一位名叫 John 的客户在周二问过退款。"
- "某个会话里提到过一次折扣。"
- "这条消息用了三段。"

## 证据标准
每一条加入任一文件的条目都必须包含：

- 对规则或模式的简明陈述
- 支撑细节
- 可见的条件、例外或限制
- 置信度标签
- 指向具体 Gmail 会话或邮件的证据来源引用

请按以下方式一致地使用置信度级别：

- `High`：在多个合格会话中反复出现，具有明显一致性
- `Medium`：有一定证据支持，但尚未达到普遍成立
- `Low`：存在可能的模式，但支持有限或存在冲突

## 冲突处理
当证据互相冲突时：

- 除非更新的或更广泛的证据明显占优，否则不要擅自决出胜者。
- 在相应的未决章节中记录该冲突。
- 保留其中的细节，既说明看起来稳定的部分，也说明仍不确定的部分。

## 规范化规则
- 使用稳定的业务术语。
- 将含义相同的重复观察合并为一条规范化的规则。
- 将相邻但不同的规则拆分为独立条目。
- 优先使用简短的事实陈述，而不是叙述性解释。
- 措辞保持中性、操作化，便于人工维护。

## 必需流程
请严格按以下顺序执行：

1. 打开并阅读 `COMPANY_BUSINESS_KNOWLEDGE.MD` 与 `EMAIL_REPLY_STYLE_GUIDE.MD`。
2. 使用 `文件状态判定` 将每个文件归入状态 A（Missing）、状态 B（Template）或状态 C（Populated）。**绝不**将状态 B 视为已完成。不要因为文件已存在就停工。
3. 在任何状态下，都要保留所有人工维护的章节以及任何明显由人工撰写的备注。
4. 使用 Gmail 连接器对 Primary 邮箱中的高信号消息建立短名单。此扫描在所有三种状态下（包括状态 C）都是**必需**的。
5. 将有价值的条目展开为完整会话，并研究其中持久的业务事实与写作模式。
6. 提取规范化的、有证据支撑的规则。
   - 对于状态 A 与状态 B：完整填充 agent 生成章节，把模板占位内容替换为真实规则。
   - 对于状态 C：将新发现的规则与现有规则合并，并在有新证据支持的情况下补强已有规则的证据。
7. 对重叠的观察进行去重。
8. 将未决或互相冲突的证据归入 `Known Unknowns`。
9. 使用简明引用更新 `Evidence Index`。
10. 验证未来的运行时起草工作不再依赖 `INIT.MD` 的任何内容。

### 停止条件与反停止规则
- 仅仅读完 `INIT.MD` 并确认目标文件存在，**不**构成完成，不得就此停手。那不是初始化。
- 不得仅因为目标文件中仍含模板文本就停手；模板状态要求执行完整的 bootstrap。
- 当文件处于状态 B 时，**不要**向用户请示是否继续，应自动继续执行。
- 在完成提取之前，**唯一**可以接受的暂停并向用户汇报的理由是：
  - Gmail 连接器本身不可用，或对每一次尝试调用都返回连接器层面的错误。
  - 目标文件存在但由于文件系统错误无法读取或写入。
  - 你发现 Human Overrides 与新证据之间存在明显冲突，必须由用户来裁决。
- 连接器或基础设施问题必须在汇报中附上具体错误、尝试执行的操作，以及仍需用户提供的内容。阻塞解除后应自动恢复继续执行。

## 质量标准
在完成之前，请确认以下所有事项：

- 业务知识文件足够详尽，足以回答客户关于服务、定价逻辑、范围、流程与升级处理的常见问题。
- 风格指南足够详尽，足以复现一致的称呼、语气、结构、排版、澄清语言、结尾习惯以及多语言行为。
- 两份文件都遵循必需的章节顺序和必需的条目 schema。
- 两份文件都清晰到足以让人工手动添加规则，而无需改动结构。
- 两份文件中的任何内容都不依赖未记录的假设。

## 完成条件
只有当以下**所有**条件都满足时，初始化才算完成：

- 两份目标文件均已存在。
- 两份文件都**不**处于状态 B（Template）。每一个必需的 agent 生成章节至少包含一条真实、有证据支撑的条目；或当邮箱在该章节确实缺乏信号时，在 `Known Unknowns` 中给出明确且合理的说明条目。
- 没有任何 agent 生成条目仍保留 `Not initialized yet`、`None yet` 等模板占位措辞。
- `Source Evidence` 字段指向具体的 Gmail 会话或邮件，而不是 `None yet`。
- Human Overrides 章节未被改动。
- 对于状态 C 的再运行，完成还额外要求：完整的邮箱扫描确实已被执行，且新发现的持久规则已被合并进文件。

如果上述任一条件未满足，则初始化**未**完成，必须继续工作。

## Bootstrap 完成后的规则
在初始化完成之后，常规的起草工作流必须忽略 `INIT.MD`。未来的草稿生成只应使用：

- `COMPANY_BUSINESS_KNOWLEDGE.MD`
- `EMAIL_REPLY_STYLE_GUIDE.MD`
- 当前正在处理的实时 Gmail 会话
